 VYRO HACKATHON CipherStack Node-Based Cascade Encryption Builder   Time Limit 1h 45m Track Frontend Difficulty Hard  What Is Cascade Encryption? Cascade encryption (also called cipher stacking) is the technique of encrypting data multiple times by passing it through a chain of different cipher algorithms in sequence. The plaintext enters the first cipher, its output feeds into the second, that output feeds into the third, and so on. The final output is a ciphertext that has been transformed by every algorithm in the chain. Because the data passes through multiple independent transformations, breaking the result requires breaking every layer — not just one. The strength of the chain comes from the diversity and order of its links. The Problem Build a node-based frontend application where users visually construct a cascade encryption pipeline. Each node represents one cipher algorithm with its own configuration. Nodes connect together to form a workflow: plaintext enters the first node, each node’s output becomes the next node’s input, and the final node produces the encrypted result.  Core Requirement Your pipeline must support a minimum of 3 cipher algorithm nodes to produce a valid output. Users should be able to stack as many nodes as they want beyond that minimum.  Your application must let the user: 1. Add cipher nodes from a library of available algorithms. 2. Configure each node independently (keys, shift values, modes, etc.). 3. Arrange nodes in a sequential pipeline (reorder, insert, remove). 4. Enter plaintext, execute the full pipeline, and view the final encrypted output. 5. See intermediate outputs — what each node received and what it produced. 6. Decrypt the final output back to the original plaintext using the same pipeline in reverse.

Encryption & Decryption Both directions are required. Your application must support two modes:  Encrypt: Plaintext enters Node 1, flows forward through every node in order, and the last node produces the final ciphertext. Decrypt: The ciphertext enters the last node first, flows backward through every node in reverse order, and each node applies its inverse operation. The first node’s output should be the original plaintext.  Every cipher you implement must have a working inverse. For example: a Caesar shift of +3 decrypts with a shift of −3, Base64 encode decrypts with decode, XOR with the same key is its own inverse, and reversing a string is its own inverse.  The Test If you encrypt any plaintext through your pipeline and then decrypt the result using the same pipeline configuration, you must get back the exact original plaintext. This is the core correctness check.  Example Cipher Algorithms You choose which ciphers to implement. Below are examples to get you started. You can use any combination of these, or implement entirely different ones. The only rule is that every algorithm must be stackable — it takes a string or byte sequence as input and produces a string or byte sequence as output, so it can chain with any other node. Pick the ones you like, add your own, or go deep on fewer algorithms with rich configuration. What matters is that they chain correctly and that each one has a working inverse for decryption.  Configurable Ciphers At least 3 of your cipher types must be configurable — meaning they accept user-defined parameters (a key, a shift value, a number of rails, etc.) that change how the algorithm behaves. Config-free ciphers like Base64, Reverse, or ROT13 can be included as extras but do not count toward this minimum.  Example Walkthrough A user builds a 3-node pipeline:  Encryption (forward): Step Node Input Output 1 Caesar (shift 3) "hello" "khoor" 2 XOR (key: "abc") "khoor" "(hex output)" 3 Vigenère (key: "secret") "(hex output)" "(final ciphertext)"  Decryption (reverse): Step Node (Inverse) Input Output 1 Vigenère Decrypt ("secret") "(final ciphertext)" "(hex output)" 2 XOR (key: "abc") "(hex output)" "khoor" 3 Caesar (shift −3) "khoor" "hello"  Encrypt → Decrypt = original plaintext. This round-trip must work for any input and any pipeline configuration. UI Expectations There is no single correct UI. You might build a drag-and-drop node graph, a vertical stack with cards, a sidebar-and-canvas layout, or something else entirely. What matters: • The pipeline is visually clear — a viewer can understand the flow at a glance.
